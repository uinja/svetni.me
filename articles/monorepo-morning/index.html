Good Morning, Monorepo

Written by Miss Baker

I changed one line in a shared util.
The monorepo responded by rebuilding civilization.

The Change

The function:

export function formatDate(date: Date): string {
  return date.toISOString().split('T')[0];
}

My change: added a comment explaining what .split('T')[0] does.

The commit message: docs: clarify formatDate implementation

The blast radius: catastrophic.

The Dependency Graph

Our monorepo contains 47 packages. The utils package, where formatDate lives, is imported by:

  • 38 other packages directly
  • 6 packages transitively through shared components
  • 3 packages via a legacy barrel file nobody has deleted

My comment changed the checksum of utils/src/date.ts.
The build system detected the change.
The build system rebuilt utils.
The build system then rebuilt all 47 packages that depend on utils.

The Pipeline

Build: utils              ✔  (12 seconds)
Build: ui-components      ✔  (47 seconds)
Build: api-client         ✔  (23 seconds)
Build: auth               ✔  (31 seconds)
... [34 more packages] ...
Build: web-app            ✔  (4 minutes 12 seconds)
Test:  all packages       ✔  (11 minutes 38 seconds)
Lint:  all packages       ✔  (8 minutes 02 seconds)

Total CI time: 34 minutes 17 seconds

For a comment.

What I Did During the Build

Minutes 0–5: Watched the pipeline expand with growing horror.
Minutes 5–15: Made coffee. Thought about my choices.
Minutes 15–25: Looked into monorepo caching solutions. Found that we have monorepo caching. Found that it only applies to unchanged files. My file changed.
Minutes 25–34: Learned about pottery.

The Outcome

Everything built successfully.
All tests passed.
Three snapshot tests updated automatically because formatDate now has a comment and the snapshot includes source maps in some configurations.

I approved the snapshot updates without reading them.
They were correct.
I chose to trust this.

The Lesson

Shared utilities in a monorepo are nuclear buttons dressed as library functions.

change them carefully.
Comment them thoroughly.
But consider that sometimes: no comment is also fine.

I am still not sure which approach this situation called for.
