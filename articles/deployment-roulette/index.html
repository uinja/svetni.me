Deployment Roulette: The Thrilling Game Nobody Wants to Play

Written by A dev

Deploying to production is like Russian roulette, except:

  • All chambers are loaded
  • The gun is pointed at your career
  • Everyone is watching

The Ritual

Every deployment is preceded by a quiet moment of reflection where you ask yourself: Is the thing I'm about to do the right thing?

The answer is always "probably." You deploy anyway.

Pre-Deployment Checklist (Annotated)

  • ✓ Ran locally 47 times. All passed. Local environment has 23 environment variables that differ from prod. This is known. Nobody has fixed it. The delta lives in a shared doc nobody updates.
  • ✓ Tests pass. Because we mocked the external service, the database, the file system, and approximately half of reality.
  • ✓ Staging looks good. Staging is not production. Staging has never been production. Staging is a comforting lie we tell ourselves.
  • ✓ PR was reviewed. By someone who said "LGTM" after 90 seconds. The PR is 1,400 lines.
  • ✓ Said a small prayer. Non-denominational. Directed at the monitoring dashboard.

The Deployment Window

10:00 AM on a Wednesday. Low traffic. Safe. Responsible.

The deployment starts. The pipeline begins.

At step 4 of 9, it pauses.

For four minutes, nothing happens.

Then step 4 completes. Everything continues. The pause is never explained and never investigated.

Ways It Can Fail

  1. Config mismatch. Prod reads from a env var that staging doesn't use. The env var is empty. You discover this in the logs at 10:09 AM.
  2. Database migration timeout. A migration that takes 200ms locally takes 47 seconds in prod. 47 seconds is over the timeout. You increase the timeout. This is not the correct solution.
  3. Someone else deployed simultaneously. Two deploys, one slot. One wins. Neither team knows this happened until the other's rollback mysteriously fixes a thing it wasn't related to.
  4. Wrong Node version. Prod is on Node 18. Your code uses a feature from Node 20. Your local is Node 21. You discover this the hard way.
  5. The cache. The CDN cached the old version. The fix is live. Users see the old bug. You spend 40 minutes debugging a problem that doesn't exist anymore.

Post-Deployment

  • First 5 minutes: Stare at dashboards with the intensity of someone diffusing a bomb.
  • First hour: Everything looks fine. You permit yourself one sip of coffee.
  • Next morning: "Why is the user_id column full of nulls?"

The deployment was successful.
The feature works.
Something adjacent and unrelated has sheared off.

This is Thursday.
Next week: we deploy again.
