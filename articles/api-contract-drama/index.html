API Contract Drama in Three Acts

Written by Miss Baker

Act I: Backend says field is optional.
Act II: Frontend assumes it always exists.
Act III: Production discovers reality.

The API Response in Question

{
  "user": {
    "id": 42,
    "name": "Ada",
    "preferences": null
  }
}

The field preferences is documented as "optional object."

The frontend code:

const theme = response.user.preferences.theme;

No null check. No optional chaining. Just optimism.

The Incident

New user registers. Preferences not yet set. preferences is null.

TypeError: Cannot read properties of null (reading 'theme')
  at UserProfile.jsx:47

Crashes for every new user on first login. This is discovered by a user who emails support, not by monitoring, because the error wasn't being logged.

The Cross-Team Summit

We hold an emergency meeting. Both teams attend.

Backend perspective: "We documented it as optional. This is correct. Null is a valid value."

Frontend perspective: "The field is present 99% of the time. Implicitly, it felt required."

Backend: "Implicit contracts are not contracts."

Frontend: "Then why is the field there at all if it can be null?"

Backend: "Historical reasons."

Silence.

The Resolution

After 94 minutes, we agreed on:

  1. Immediate fix: Frontend adds null-safe access everywhere. (preferences?.theme)
  2. Medium-term: Backend will return a default preferences object instead of null.
  3. Long-term: We will adopt an OpenAPI schema and validate responses against it.

Status of these items, twelve weeks later:

  1. Done ✔
  2. In review for six weeks. Backend team changed priorities.
  3. In the backlog. Tagged enhancement.

Lessons, Filed

  • Optional does not mean "will never be null."
  • "Will never be null in practice" is not in the spec and is not binding.
  • Shared schemas are worth writing. They are also worth enforcing.
  • New users will always find the cases that long-term users don't.

The preferences field now returns {} by default.
The null check is still in the frontend.
Trust, once lost, is defensive-coded in.
